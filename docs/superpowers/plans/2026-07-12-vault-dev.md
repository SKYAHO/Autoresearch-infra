# HashiCorp Vault dev 도입 구현 계획

> 설계: `docs/superpowers/specs/2026-07-12-vault-dev-design.md`
> Issue: (발행 예정 — 단계별 3개)

## 1단계: dev root — KMS auto-unseal 기반

1. `terraform/envs/dev/vault.tf` 신규:
   - `google_kms_key_ring` `autoresearch-dev-vault` (asia-northeast3)
   - `google_kms_crypto_key` `vault-unseal` (rotation 90d)
   - GSA `autoresearch-dev-vault` + key 단위
     `roles/cloudkms.cryptoKeyEncrypterDecrypter`
   - WI 바인딩: `serviceAccount:<project>.svc.id.goog[vault/vault]`
2. 문서: TERRAFORM_DEV.md 파일 트리·IAM 표, INFRASTRUCTURE_SUMMARY.md IAM/WI
3. 검증: fmt/validate → CI plan (add만, change/destroy 0) → merge → apply

## 2단계: vault-k8s admin root — 설치와 경계

1. `terraform/admin/vault-k8s/` 신규 (argocd-k8s 구조 복제):
   - backend GCS prefix `admin/vault-k8s`, providers, variables
   - namespace `vault`(PSS baseline 라벨, prevent_destroy), KSA `vault`
     (GSA annotation)
   - NetworkPolicy ingress/egress (설계 표 그대로 — services CIDR,
     metadata 987/988 포함)
   - helm_release `vault` (chart 버전 pin, atomic, values 파일:
     standalone raft 1 replica, PVC 10Gi, gcpckms seal 블록, ClusterIP,
     resources requests/limits)
2. runbook `docs/VAULT_OPERATIONS_RUNBOOK.md`:
   - port-forward 접근, `vault operator init`(recovery keys/root token
     취급 절차 — 출력물 보관·revoke), status 확인, audit device 활성화,
     장애 시 복구(재기동 auto-unseal), 롤백
3. README(`terraform/admin/vault-k8s/README.md`): 책임 범위, 사용 방법,
   재해복구 순서
4. 검증: 설계 문서의 검증 기준 5개 전부 + 결과를 이슈에 기록

## 3단계(후속 이슈): 기능 활성화

1. Kubernetes auth method + KV v2 engine mount, 시범 secret 1개(더미 값)
2. policy 최소 권한 예시 1개
3. TLS 활성화 및 ESO 연동 검토는 별도 설계 후
